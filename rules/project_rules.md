# 项目规则

## 1. 项目基本信息

- **项目名称**: 复刻土豆兄弟 (Brotato-like)
- **项目类型**: 2D Rogue-like 独立游戏
- **游戏引擎**: Godot Engine 4.5
- **开发语言**: C#

## 2. 项目结构规范

### 2.1 目录结构

```
brotato-my/
├── assets/              # 游戏资源目录
│   ├── character/       # 角色资源
│   ├── environment/     # 环境资源

│   ├── enemies/         # 敌人资源
│   ├── ui/              # UI资源
│   ├── audio/           # 音频资源
│   └── fonts/           # 字体资源
├── scenes/              # 场景文件目录
```

### 2.2 文件命名规范

- **场景文件 (.tscn)**: 使用 PascalCase，如 `Player.tscn`, `MainMenu.tscn`
- **脚本文件 (.cs)**: 使用 PascalCase，如 `Player.cs`, `EnemySpawner.cs`
- **资源文件**: 使用 snake_case，如 `hero_guangfa.png`, `background_music.wav`
- **目录名称**: 使用 snake_case，如 `character_assets`, `ui_scenes`

## 3. Godot 引擎特定规则

### 3.1 脚本编写规范

- 类名使用 PascalCase，与文件名保持一致
- 变量和函数使用 snake_case
- 常量使用 ALL_CAPS
- 使用类型注解提高代码可读性和性能
- 为公共函数和属性添加文档注释
- **命名空间**: 采用 **混合策略 (Hybrid Strategy)**：
  - **全局命名空间 (Global Namespace)**: 适用于高频使用的核心类，如工具 (`Log`, `ObjectPool`)、组件 (`HealthComponent`)、核心实体 (`Player`)。避免繁琐的 `using`。
  - **特定命名空间**:
    - **测试代码**: 必须使用 `namespace BrotatoMy.Test`，防止测试类污染全局智能提示。
    - **第三方库**: 必须封装在独立命名空间中。
    - **模块化子系统**: 仅当确实存在命名冲突风险时使用。

### 3.2 节点组织规范

- 使用有意义的节点名称
- 合理使用节点分组 (Groups)
- 优先使用组合而非继承
- 避免过度嵌套节点

### 3.3 资源管理规范

- 所有资源必须通过 `preload()` 或 `load()` 加载
- 大型资源应使用异步加载
- 资源使用后及时释放，避免内存泄漏
- 使用资源池管理频繁创建和销毁的资源

## 4. 资源使用规则

### 4.1 解包游戏资源使用

- **解包游戏路径**: `E:\Godot\Games\steam解包游戏\土豆兄弟`
- **资源引用方式**:
  1. 仅允许引用解包资源作为参考和学习
  2. 禁止直接复制解包资源到项目中
  3. 所有最终使用的资源必须进行修改或重新创建，确保不侵犯原游戏版权
  4. 解包资源仅用于：
     - 了解游戏机制和设计思路
     - 参考美术风格和动画效果
     - 分析游戏平衡数据

### 4.2 资源创建规范

- 美术资源：优先使用原创或开源资源
- 音频资源：使用免费或购买的版权音乐和音效
- UI 资源：保持与游戏风格一致的设计
- 所有资源必须符合项目的整体风格

## 5. 代码规范

### 5.1 GDScript 编码规范

- 缩进使用 4 个空格
- 每行不超过 100 个字符
- 使用空行分隔不同逻辑块
- 避免使用全局变量，优先使用信号和事件
- 合理使用注释，解释复杂逻辑

### 5.2 设计模式

- 采用组件化设计
- 使用信号系统进行节点间通信
- 实现单例模式管理全局状态
- 使用工厂模式创建实体

## 6. 版本控制规则

### 6.1 Git 提交规范

- 提交信息使用中文，格式：`[类型] 描述`
- 类型包括：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`docs`（文档）、`style`（格式）、`test`（测试）
- 每个提交只包含一个功能或修复
- 提交前确保代码可以正常运行

### 6.2 忽略文件

- 已配置 `.gdignore` 和 `.gitignore`
- 忽略 Godot 编辑器生成的临时文件
- 忽略编译产物和缓存文件

## 7. 开发流程

### 7.1 任务管理

- 使用项目管理工具跟踪任务
- 每个任务对应一个分支
- 完成任务后提交 Pull Request 进行代码审查

### 7.2 测试规范

- 定期进行功能测试
- 记录和跟踪 bug
- 实现自动化测试（如有必要）

## 8. 性能优化

### 8.1 渲染优化

- 使用合适的纹理尺寸
- 合理设置精灵的 `draw_behind_parent` 属性
- 避免过度使用透明效果
- 使用批处理减少 draw call

### 8.2 脚本优化

- 避免在 `_process()` 中执行复杂计算
- 使用 `_physics_process()` 处理物理相关逻辑
- 合理使用定时器和延迟调用
- 优化循环和条件判断
