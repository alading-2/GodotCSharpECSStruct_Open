# 深度思考：对象池重置策略 (Reset Strategy)

## 核心问题
**"对象重置 (Reset) 究竟应该发生在 `归还时 (Release)` 还是 `取出时 (Acquire)`？"**

这是一个贯穿游戏引擎设计和高性能编程的经典问题。在 C# + Godot + ECS 的架构下，这个问题的答案并非非黑即白，而是取决于**由谁持有引用**以及**数据的生命周期**。

## 结论先行

在我们的架构中，推荐采用 **混合策略 (Hybrid Strategy)**：

1.  **归还时 (OnRelease)**：执行 **"清理 (Cleanup)"**。
    *   **核心目标**：断开引用，防止内存泄漏，防止脏数据影响池内其他对象。
    *   **关键词**：`Clear`, `Nullify`, `Unsubscribe`, `Stop`。
2.  **取出时 (OnAcquire)**：执行 **"初始化 (Initialize)"**。
    *   **核心目标**：注入新身份，设置新状态。
    *   **关键词**：`Load`, `Set`, `Start`, `Spawn`。

---

## 1. 归还时重置 (Reset on Release) —— "清洗"

这是**安全防线**。如果我们不在这里做清理，就会留下"尸体"和"幽灵"。

### 必须做的 (Mandatory)
*   **断开引用 (Reference Breaking)**：
    *   如果 `Bullet` 持有 `Target (Enemy)` 引用，而 `Enemy` 被销毁了。如果 `Bullet` 回池时不置空 `Target`，垃圾回收器 (GC) 就无法回收那个 `Enemy`。
    *   **后果**：严重的内存泄漏 (Memory Leak)。
*   **解绑事件 (Event Unsubscription)**：
    *   如果 `HealthComponent` 订阅了全局 `EventBus`，回池后不解绑。
    *   **后果**：当 Global Event 触发时，池子里"沉睡"的对象会被唤醒执行逻辑，导致报错或逻辑混乱（"诈尸"）。
*   **停止逻辑 (Stop Coroutines/Tweens/Timers)**：
    *   如果有正在运行的 Timer 或 Tween，必须停止。
    *   **后果**：对象在池子里突然发生位移，或者在后台消耗 CPU。

### 可以不做，但推荐做的 (Optional but Recommended)
*   **数值归零**：将 HP 重置为 MaxHP，Score 重置为 0。
    *   **思考**：虽然 `Spawn` 时通常会覆盖这些值，但让池内对象保持"干净"状态有助于调试。如果 Spawn 逻辑有漏洞忘记覆盖某个值，"默认值"总比"上一次的残留值"要安全。

---

## 2. 取出时重置 (Reset on Acquire) —— "赋予新生"

这是**业务起点**。有些事情只能在这里做，因为只有这里才知道"我是谁"。

### 必须做的 (Mandatory)
*   **身份注入 (Identity Injection)**：
    *   `Data.LoadFromResource(resource)`。只有在取出时，我们才知道这个 Enemy 是 "小怪A" 还是 "Boss B"。
*   **视觉重置 (Visual Reset)**：
    *   播放 `Spawn` 动画，重置 Sprite 颜色（如果之前被打红了）。
*   **特异性初始化**：
    *   设置初始位置、初始旋转、初始速度。

---

## 3. 为什么我们的 Component Reset 放在 Unregister 时？

在 `EntityManager.UnregisterEntity` 的流程中：

```csharp
// 1. 先让组件有机会清理
UnregisterComponents(entity); 
    -> Component.OnComponentUnregistered() 
    -> Component.OnComponentReset() 
        -> 此时 Entity.Data 还在，可以读取 LastHitInfo, 移除 Buff 效果等

// 2. 再彻底抹除实体痕迹
Entity.Events.Clear();
Entity.Data.Clear();
```

**深度原因**：
Component 是寄生在 Entity 上的。
*   **当 Entity 归还池时 (Release)**：Component 必须随之"休眠"。如果不清理 Component（解绑事件、清除 Buff），这个 Component 就会成为"幽灵"。
*   **当 Entity 从池取出时 (Acquire)**：Entity 获得新的 Resource，Data 被重新填充。此时 Component 再次 `OnComponentRegistered`，读取**新**的数据，重新开始工作。

所以，Component 的 `Reset` 本质上是 **"休眠前的清理"**，必须在 Entity 失去意识（Data/Events Clear）之前完成。

## 4. 总结对比

| 阶段 | 动作名称 | 职责 | 这种不做会有什么严重后果？ |
| :--- | :--- | :--- | :--- |
| **归还时 (Release)** | **Cleanup (清理)** | 擦除上一世的痕迹 | 内存泄漏 (引用未断)、诈尸 (事件未解绑)、脏数据污染 |
| **取出时 (Acquire)** | **Setup (装填)** | 赋予这一世的属性 | 对象是空的/错误的（数值不对、外形不对） |

### 最佳实践口诀

> **"走的时候带走垃圾（断引用、清事件），来的时候穿好新衣（加载配置、重置状态）。"**

*   **IPoolable.OnPoolRelease**: 负责 "忘却"。
*   **IPoolable.OnPoolAcquire**: 负责 "醒来"。
