# 深度思考：数据驱动架构 (Data-Driven Architecture)

## 核心哲学：逻辑与数据的彻底分离

在传统的面向对象编程 (OOP) 中，对象通常同时包含**行为 (Method)** 和**状态 (Field)**。
而在我们的架构中，我们推崇 **"逻辑 (Component)"** 与 **"数据 (Data Container)"** 的分离。

```csharp
// ❌ 传统 OOP (状态与逻辑耦合)
class HealthComponent {
    private float _hp; // 状态
    public void TakeDamage(float amount) { _hp -= amount; } // 逻辑
}

// ✅ 数据驱动 (状态归 Data，Component 只管逻辑)
class HealthComponent {
    // 无状态
    public void TakeDamage(float amount) { 
        var hp = data.Get<float>("HP");
        data.Set("HP", hp - amount); 
    }
}
```

## 为什么选择这种“透传”模式？

您提到的 `public Type Name => _data.Get<Type>(Key);` 模式，不仅仅是语法糖，它代表了一种架构思想的转变。

### 1. 单一数据源 (Single Source of Truth)
*   **问题**：如果 Component 内部缓存了 `_privateHp`，那么 `Data["HP"]` 和 `_privateHp` 就可能不同步。哪个才是真的？
*   **解决**：彻底放弃 Component 内部状态。`Data` 是唯一的真理。Component 只是 `Data` 的"操作员"或"视图"。

### 2. 极简的生命周期管理 (Stateless = Painless)
*   **问题**：对象池复用最头疼的就是 Reset。如果 Component 有 10 个私有字段，Reset 就要写 10 行。漏写一个就是 Bug。
*   **解决**：因为 Component 是无状态的，它需要的状态都在 `Entity.Data` 里。
    *   Entity 销毁/回池时，`Data.Clear()` 一键清空所有数据。
    *   下次 Spawn 时，`Data.LoadFromResource()` 注入新数据。
    *   **Component 不需要做任何数据重置！** (除了 Timer/Tween 等瞬时过程状态)。

### 3. 极致的灵活性 (Runtime Flexibility)
*   **场景**：策划想给某个单位临时加个 "Shield" 属性，但代码里没定义这个字段。
*   **优势**：直接 `data.Add("Shield", 100)` 即可。其他 Component 可以立即读取到，无需修改类定义或重编译。

## 我们的架构定位：实用主义 ECS (Pragmatic ECS)

*   **纯 ECS (Unity DOTS)**：Entity 是 ID，Component 是 Struct (纯数据)，System 是逻辑。性能极致，但开发繁琐。
*   **传统 Unity/Godot**：MonoBehaviour/Node 既有数据又有逻辑。开发快，但复用难，序列化难。
*   **我们的架构**：
    *   **Entity** = Godot Node (利用引擎优势)
    *   **Data** = Dictionary (动态数据容器)
    *   **Component** = Logic Node (无状态逻辑单元)

这种架构保留了 ECS "数据逻辑分离" 的优点，同时保留了 Godot "节点树" 的易用性。

## 总结

> **"Data is the King, Component is the Worker."**

*   **Data**：负责存活下去（序列化、持久化、跨场景传递）。
*   **Component**：负责干活（计算、移动、渲染）。干完活（Entity 销毁）就可以直接扔掉（回池），因为它不带走一片云彩。
