## 1. 核心设计哲学

> [!IMPORTANT]
> **技能系统完全遵循项目 ECS 框架原则**，不引入任何外部框架的独立类概念。
> **核心思想**：Data 驱动状态，Event 驱动流程，System 驱动逻辑。

**架构一致性矩阵**：

| ECS 原则 | 技能系统实现 |
|:---|:---|
| **Scene 即 Entity** | `AbilityEntity` 继承 `Node` 实现 `IEntity`，归还对象池管理 |
| **Data 唯一数据源** | 所有技能属性（CD、伤害、范围）均通过 `DataKey` 存储，严禁组件私有状态 |
| **Component 无状态** | 组件仅负责响应事件和读写 `Data`，不持有运行时数据 |
| **EntityManager 入口** | 通过 `EntityManager.AddAbility` 创建，由关系管理器维护 Owner 关系 |
| **事件驱动通信** | 组件间解耦，通过 `Events.Emit()` 进行 "请求-响应"式交互 |

## 2. 核心架构：Trigger / Cast / Execute 三层分离

| 阶段 | 英文 | 职责 | 核心组件/系统 |
|:---|:---|:---|:---|
| **1. 触发层** | **Trigger** | 判断"什么时候该放"。只发出请求，不扣资源。 | `TriggerComponent` / `ActiveSkillInputComponent` |
| **2. 施法层** | **Cast** | 判断"能不能放"。冷却、消耗、目标验证。 | `AbilitySystem` (Static) |
| **3. 执行层** | **Execute** | 处理"怎么放"。生成子弹、造成伤害、应用 Buff。 | `AbilityExecutorRegistry` -> 具体 Executor |

### 2.1 触发层 (Trigger)

- **职责**：产生"施法请求"。
- **来源**：玩家输入（`ActiveSkillInputComponent`）、AI 决策、全局事件、周期性定时器（`TriggerComponent`）。
- **关键点**：触发层 **不应该** 直接修改冷却或消耗资源，它只是一个 "Intention" (意图)。

### 2.2 施法层 (Cast)

`AbilitySystem.HandleTryTrigger()` 收到 `TryTrigger` 事件后，进入 `TryTriggerAbilityWithContext()` 流水线：

1. **CanUse** — 发送 `CheckCanUse` 事件，按 `AbilityCheckPhase` 优先级（Cooldown → Cost → TargetValidity → Custom）依次检查
2. **SelectTargets** — 发送 `SelectTargets` 事件，`AbilityTargetSelectionComponent` 填充 `CastContext.Targets`
3. **TargetResolution**（统一目标解析阶段）— 根据 `AbilityTargetSelection` 分流：
   - **Entity** — 必须有目标，否则中止（避免空放浪费资源）
   - **Point** — 如果还没有位置，调用 `RequestPlayerTargeting()` 进入异步瞄准，返回 `WaitingForTarget`
   - **EntityOrPoint** — 先尝试 Entity 自动索敌，未命中则回退 Point 异步瞄准
   - **None** — 直接通过
4. **ConsumeCharge** — 消耗充能
5. **StartCooldown** — 启动冷却（周期性技能跳过）
6. **ConsumeCost** — 消耗魔法/能量等
7. **Execute** — 通过 `AbilityExecutorRegistry` 执行具体效果

> 异步瞄准完成后，`TargetingManager` 回调 `AbilitySystem.ResumeAfterTargeting(context)` 重新走完整流水线（会再次检查 CanUse，因为瞄准期间时间已过）。

### 2.3 执行层 (Execute)

- **独立性**：执行逻辑不写在 `EntityManager`，通过 `AbilityExecutorRegistry` 注册策略模式执行器。
- **无状态**：执行器只依赖传入的 `CastContext` 进行操作。
- **多态性**：每个技能可以有完全不同的执行逻辑。

## 3. 目标选择设计

### 3.1 统一流水线，内部分流

借鉴 War3/DOTA2/UE5 GAS 的行业标准，所有目标类型统一通过 `TryTrigger` 事件入口（`AbilitySystem.HandleTryTrigger`），调用者无需关心内部是同步还是异步：

| 类型 | 时序模型 | 流水线内部处理 | 返回值 |
|:---|:---|:---|:---|
| **Entity** (单位目标) | **同步** | `AbilityTargetSelectionComponent` 空间查询 → 目标验证 | `Success` / `Failed` |
| **Point** (位置目标) | **异步** | `RequestPlayerTargeting()` → `TargetingManager` 接管 | `WaitingForTarget` |
| **EntityOrPoint** | **先同步后异步** | 先尝试 Entity 索敌，未命中则回退 Point 瞄准 | `Success` / `WaitingForTarget` |
| **None** (无目标) | **同步** | 跳过目标选择，直接执行 | `Success` / `Failed` |

**设计原则**：Entity 目标同帧完成空间查询，Point 目标跨帧等待玩家输入——时序本质不同，但对调用者透明。流水线通过 `TriggerResult` 枚举统一表达结果（Success / Failed / WaitingForTarget）。

### 3.2 技能分类与手柄适配

| 分类 | War3 原型 | 手柄操作 |
|:---|:---|:---|
| **无目标 (Instant)** | 雷霆一击、天神下凡 | 按 X 键直接释放 |
| **单位目标 (Unit Target)** | 风暴之锤、死亡缠绕 | 按 X 键 → 系统自动索敌（最近/最高威胁） |
| **位置目标 (Point Target)** | 暴风雪、烈焰风暴 | 按 X 键 → 进入瞄准模式 → 右摇杆控制指示器 → X 确认 / B 取消 |

### 3.3 瞄准系统 (TargetingSystem)

Point 类型技能的异步子系统，由 `AbilitySystem.RequestPlayerTargeting()` 触发：

```
StartTargeting → 生成 TargetingIndicatorEntity → 右摇杆移动（射程限制）→ X 确认 / B 取消 → EndTargeting
```

- **TargetingManager**：瞄准状态机，管理异步流程
- **TargetingIndicatorEntity**：可视化施法点标记（War3 "马甲"模式）
- **TargetingIndicatorControlComponent**：右摇杆输入、射程限制、确认/取消处理

## 4. 组件职责与通信

组件不直接相互调用，而是通过事件 + 上下文模式协作（`EventContext` + `CastContext.ResponseContext`）。

### 组件职责矩阵

| 组件 | 核心职责 | 监听事件 | 发送事件 |
|:---|:---|:---|:---|
| `TriggerComponent` | 触发源管理 (手动/自动/事件/周期) | 全局事件 | `TryTrigger` |
| `CooldownComponent` | 冷却计时与检查 | `CheckCanUse` / `StartCooldown` / `ResetCooldown` | `Ready` |
| `ChargeComponent` | 充能次数管理 | `CheckCanUse` / `ConsumeCharge` | `ChargeRestored` |
| `CostComponent` | 资源消耗管理 (魔法/能量/生命值) | `CheckCanUse` / `ConsumeCost` | — |
| `AbilityTargetSelectionComponent` | Entity 类型目标解析 | `SelectTargets` | — |

### 通信流程图

```mermaid
sequenceDiagram
    participant Input as 触发源(Input/AI)
    participant Sys as AbilitySystem
    participant E as AbilityEntity.Events
    participant Cmp as Components(CD/Charge/Cost)
    participant Exec as AbilityExecutorRegistry

    Input->>E: Emit(TryTrigger, TryTriggerEventData(context))
    E->>Sys: HandleTryTrigger(context)
    Sys->>E: Emit(CheckCanUse)
    E->>Cmp: 按 AbilityCheckPhase 优先级检查
    Cmp-->>Sys: EventContext.Success / Failed

    alt 检查通过
        Sys->>E: Emit(SelectTargets)
        Note right of Sys: Entity类型: 同帧解析<br/>Point类型: 已由TargetingManager预填充
        Sys->>Sys: 目标验证(Entity类型必须有目标)
        Sys->>E: Emit(ConsumeCharge)
        Sys->>E: Emit(StartCooldown)
        Sys->>E: Emit(ConsumeCost)
        Sys->>Exec: ExecuteAbilityEffects(context)
    end
```

## 5. 主动技能输入设计

### 5.1 设计决策

> **核心洞察**：Survivor 类游戏应尽量减少主动操作。采用 **单槽位 + LB/RB 切换** 方案。

| 按键 | 功能 |
|:---|:---|
| **LB / RB** | 切换当前选中的主动技能（左/右循环） |
| **X** | 释放当前选中技能 |

### 5.2 输入流程

`ActiveSkillInputComponent` 挂载到 `PlayerEntity`，每帧轮询输入：

1. **LB/RB** → `CycleActiveAbility()` → 更新 `DataKey.CurrentActiveAbilityIndex` → 发送 UI 事件
2. **X** → `TryUseCurrentActiveAbility()` → 发射 `TryTrigger` 事件
   - 调用者无需关心目标类型，流水线内部自动判断同步执行或异步瞄准
   - `AbilitySystem` 将结果写入 `CastContext.ResponseContext`，调用者读取 `TriggerResult`（`Success` / `Failed` / `WaitingForTarget`）

### 5.3 技能筛选

通过 `EntityManager.GetManualAbilities()` 统一筛选（输入组件和 UI 共用）：
- `AbilityType != Passive`
- `AbilityTriggerMode.HasFlag(Manual)`

结果缓存在 `ActiveSkillInputComponent` 中，仅在技能增删事件时刷新，避免每帧 LINQ 分配。

### 6.1 架构

```
ActiveSkillBarUI (HBoxContainer) — 4 槽位固定布局
├── ActiveSkillSlotUI × 4
│   ├── Background
│   ├── SkillIcon
```

### 6.2 事件驱动更新

| 事件 | 响应 |
|:---|:---|
| `Ability.Added` / `Ability.Removed` | 刷新槽位绑定 |
| `UI.ActiveSkillSelected` | 切换高亮 |

冷却进度通过 `_Process` 轮询实现平滑动画（非事件驱动）。

## 7. 数据结构设计 (DataKeys)

不存在 `AbilityConfig` 类，所有配置扁平化存储于 `Data` 中。

| 分类 | DataKey | 说明 |
|:---|:---|:---|
| **触发** | `AbilityTriggerMode` | Manual / OnEvent / Periodic / Permanent (位掩码) |
| **目标** | `AbilityTargetSelection` | None / Entity / Point / EntityOrPoint |
| | `AbilityTargetGeometry` | Single / Circle / Box / Line / Cone / Chain / Global |
| | `AbilityTargetTeamFilter` | Friendly / Enemy / Neutral / Self |
| | `AbilityTargetSorting` | Nearest / Farthest / HighestThreat / LowestHp 等 |
| | `AbilityCastRange` | 施法距离（0=无限制；索敌/瞄准射程均用此值） |
| | `AbilityEffectRadius` | 效果半径（圆形/扇形 AOE ；冲刺=位移距离） |
| | `AbilityEffectLength` | 效果长度（矩形/线形 AOE） |
| | `AbilityEffectWidth` | 效果宽度（矩形/线形 AOE） |
| **冷却** | `AbilityCooldown` | 基础冷却时间 |
| **充能** | `AbilityMaxCharges` | 最大充能数 |
| | `AbilityChargeTime` | 充能回复时间 (<0 不自动回复) |
| **消耗** | `AbilityCostType` | None / Mana / Energy / Ammo / Health |
| | `AbilityCostAmount` | 消耗数量 |

## 8. 设计问答 (Q&A)

### Q1: 为什么执行逻辑不放在 EntityManager?
`EntityManager` 的职责是 **生命周期管理**。执行逻辑属于 **业务行为**，放入管理器会导致上帝类。

### Q2: 为什么 TriggerComponent 不直接扣冷却?
为了支持 "试图释放但失败" 的情况（被沉默、目标丢失等）。触发层只表达意图，施法层决定能否执行。

### Q3: 为什么 Entity 和 Point 目标统一入口但内部分流?
Entity 目标是同步操作，Point 目标是异步操作——时序本质不同，但调用者不应关心这个区别。所有目标类型统一走 `TryTrigger` 事件入口（`AbilitySystem.HandleTryTrigger`），流水线内部通过 `TriggerResult` 枚举表达结果（Success / Failed / WaitingForTarget）。

### Q4: EntityOrPoint 混合模式如何实现?
参考 DOTA2：`AbilityTargetSelectionComponent` 先尝试 Entity 空间查询，如果命中则同步执行；未命中则由 `AbilitySystem` 自动回退到 Point 异步瞄准模式。
