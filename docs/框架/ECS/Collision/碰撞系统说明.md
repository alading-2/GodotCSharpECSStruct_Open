# 碰撞系统说明

## 一、系统定位

当前项目的碰撞系统分成三层：

- **模板资源层**：`Src/ECS/Base/Component/Presets/Collision/`
- **统一事件桥接层**：`CollisionComponent`、`HurtboxComponent`
- **业务消费层**：`ContactDamageComponent`、`EntityMovementComponent`、拾取/命中类组件

核心原则是把“谁在碰撞”“谁被碰到”“碰到后做什么”拆开，不再用一套通用碰撞事件承载所有语义。

## 二、当前架构

### 2.1 视觉体碰撞

适用于子弹、特效等以 `Area2D` 为主体的实体。

处理链路：

- `CollisionComponent`
- `GameEventType.Collision.CollisionEntered / CollisionExited`
- `EntityMovementComponent`

`CollisionComponent` 现在只负责 **Entity 根节点为 `Area2D` 时的视觉体碰撞桥接**，不再携带 `CollisionType`。

### 2.2 受击区碰撞

适用于角色的受击区、拾取区等业务感应区。

处理链路：

- `HurtboxComponent`
- `GameEventType.Collision.HurtboxEntered / HurtboxExited`
- `ContactDamageComponent`

`HurtboxComponent` 现在本身就是 `Area2D`，在 Entity 场景里直接配置 `CollisionShape2D`、`collision_layer` 和 `collision_mask`。

### 2.3 模板资源

碰撞预设模板仍保留少量 `Area2D` 预设资源，作为可复用的碰撞配置参考。


这些模板只用于复用碰撞形状与 layer/mask 配置，不再作为独立接入语义。

这些预设模板只作为复用资源存在，不再承担任何旧式容器接入职责。

## 三、节点结构约定

### 3.1 视觉体实体

```text
EffectEntity / BulletEntity (Area2D)
  ├─ CollisionComponent
  └─ VisualRoot / CollisionShape2D
```

### 3.2 角色实体

```text
Player / Enemy (CharacterBody2D)
  ├─ Component
  │   ├─ CollisionComponent
  │   ├─ HurtboxComponent
  │   └─ ContactDamageComponent（可选）
  ├─ CollisionShape2D          ← 根物理体
  └─ VisualRoot
      └─ CollisionShape2D      ← 视觉碰撞模板
```

### 3.3 Hurtbox 结构

```text
HurtboxComponent (Area2D)
  └─ CollisionShape2D
```

不再需要 `Collision/` 容器，也不再依赖节点名 `*Hurtbox*` 自动识别。

## 四、事件语义

### 4.1 视觉体碰撞事件

- `CollisionEntered(Source, Target)`
- `CollisionExited(Source, Target)`

这里只表示“Entity 根节点这个 Area2D 触发了碰撞”。

### 4.2 受击区事件

- `HurtboxEntered(Source, Hurtbox, Target, TargetEntity)`
- `HurtboxExited(Source, Hurtbox, Target, TargetEntity)`

这里的 `Hurtbox` 就是当前 `HurtboxComponent` 自身。

## 五、与其它系统的分工

### 5.1 `ContactDamageComponent`

只消费 `HurtboxEntered / HurtboxExited`，负责持续接触伤害和节流计时器。

### 5.2 `EntityMovementComponent`

只在非默认运动模式下消费视觉体碰撞：

- `Area2D` 路径：监听 `CollisionEntered`
- `CharacterBody2D` 路径：监听 `MoveAndSlide()` 的 slide collision

如果 `DestroyOnCollision = true`，则先发 `MovementCollision`，再完成移动并销毁实体。

### 5.3 `SpriteFramesGenerator`

只负责生成视觉场景中的碰撞模板，不负责运行时事件桥接。

### 5.4 `ResourceGenerator`

当前只负责资源路径生成，不再生成碰撞注册表。

## 六、对象池兼容约定（2026-04）

凡是**根节点本身参与物理世界**的池化实体（`CollisionObject2D`，如 `Area2D`、`CharacterBody2D`），都必须遵循统一的出池/回池时序：

- 回池时先禁用处理与显示
- 根节点是碰撞类型时，先停放到远离战场的泊车位，再脱树
- 出池时先挂回场景树，但保持碰撞关闭
- `EntityManager.Spawn` 先设置位置/旋转、`ForceUpdateTransform()`、注册组件
- 最后由 `pool.Activate()` 统一恢复处理与碰撞
- `CharacterBody2D` 在 `Activate()` 之后再 `CallDeferred(MoveAndSlide)`，同步物理代理

这套顺序的目标只有一个：**避免复用对象在旧死亡坐标或半初始化状态下重新参与宽相计算，触发伪 `entered` 事件。**

## 七、使用建议

- 视觉体碰撞用 `CollisionComponent`
- 受击区碰撞用 `HurtboxComponent`
- 受击区直接消费 `HurtboxEntered / HurtboxExited`
- 调整受击区形状时，直接改 Entity 场景中的 `HurtboxComponent`
- 池化的 `Area2D` / `CharacterBody2D` 不要只靠 `Monitoring` 或 `disabled` 模拟离场，统一走对象池兼容方案
