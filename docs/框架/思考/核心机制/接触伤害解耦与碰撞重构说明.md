# 碰撞与接触伤害重构文档

> 核心目标：解耦“我是谁，我碰到了谁”与“我碰到了谁后该发生什么逻辑”

## 原 HurtComponent 痛点

曾经我们使用一个庞大的 `HurtComponent` 来处理实体间身体接触引发的伤害。这导致了以下问题：
1. **职责过载**：既负责维护 `Area2D` 的碰撞检测，也负责管理给碰到的实体上 DOT（持续伤害）的计时器，并且自己就发送了具体的伤害请求。
2. **场景耦合严重**：原 `HurtComponent.tscn` 强行将组件和默认碰撞圆绑在一起，每个新怪物都要实例化这个场景并覆盖其碰撞层掩码设置，极易引发手误和 Bug（历史上的“幽灵碰撞”）。
3. **扩展困难**：如果要给怪物加一个“触碰后爆炸”的需求，由于 `HurtComponent` 里写死了触碰后只造成持续接触伤害的逻辑，这使得扩展不同触碰反馈十分痛苦。

## 新架构设计 (Event-Driven)

为了彻底解决上述痛点，我们将职责一分为二，通过 **Entity 局部事件总线 (`EventBus`)** 进行通讯。

### 1. 情报发送机 —— `CollisionSensorComponent`
这就是一个纯粹的“眼睛和皮肤”。它继承自原生 `Area2D`，挂载着物理碰撞形状 `CollisionShape2D`。
- **唯一职责**：向组件所在的 Entity 事件总线广播标准化物理事件：
    - `GameEventType_Physics.CollisionEntered`：有 Body 或 Area 进入感应区。
    - `GameEventType_Physics.CollisionExited`：有 Body 或 Area 离开感应区。
- **底层安全**：在组件注销时，使用 `SetDeferred` 彻底关闭 `Monitoring` 和 `Monitorable`，确保对象进入缓存池后不再与物理空间发生任何干涉（因为仅设置实体的 `ProcessMode = Disabled` 无法关闭 Godot 底层的碰撞监听）。
- **重点**：此组件绝不处理任何伤害、计算穿透等业务逻辑！

### 2. 接触伤害分发处理 —— `ContactDamageComponent`
这是一个没有场景节点外壳的纯 `Node` (也可以只是逻辑接口，但这里为了方便挂载选作 Node)。
- **职责**：监听并处理 `CollisionEntered` / `CollisionExited` 事件。
- 只有对方是敌对阵营，且满足某些伤害规则时，它才会为其维护一个循环 `Timer`。
- `Timer` 触发时，向 `DamageService` 抛出 `DamageInfo` 实现具体的接触伤害扣血。
- 如果以后需要触碰爆炸，可以再写一个 `ContactExplosionComponent` 同样监听物理事件做处理，互不干扰！

## 如何为一个实体配置接触伤害？

现在的配置方式完全符合 Godot "组合优于继承" 的场景搭建思想：

1. 打开目标实体的主场景（例如 `EnemyEntity.tscn`）。
2. 在代表了 ECS 组件层级的 `Components` 节点下：
    - 新建一个 `Area2D` 节点，命名为 **CollisionSensorComponent**，挂载对应的 C# 脚本。
    - 在该 Sensor 节点下，添加原生 `CollisionShape2D`，在编辑器中直观地设置碰撞范围和形状。
    - **【关键】** 在编辑器的 `CollisionObject2D` 面板里，明确设置该 Sensor 的 `Layer`（我是谁） 和 `Mask`（我要检测谁）。
    - 接着，在 `Components` 节点下，新建一个普通 `Node`，命名为 **ContactDamageComponent**，并挂载对应脚本。
    
> 💡 总结：不需要再去继承或实例化什么 `HurtComponent.tscn`，直接像搭积木一样添加 Sensor 和对应的 Damage 处理脚本，再利用 Godot 编辑器可视化的好处设置 Layer 即可。
