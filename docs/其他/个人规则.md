# 个人规则 (Personal Rules)

> 本文档记录了本项目的核心开发原则、代码规范以及自动化工作流要求。
> 旨在确保代码的高性能、可维护性以及开发体验的一致性。

---

## 1. 核心开发原则

- **组件化优先 (Composition over Inheritance)**:
  - 所有的功能模块应尽可能封装为独立的 `Node` 组件（如 `HealthComponent`, `VelocityComponent`）。
  - 实体（Entity）通过组合这些组件来获得能力。
- **数据驱动 (Data-Driven)**:
  - 静态配置数据使用 Godot 的 `Resource` (.tres 文件)。
  - 运行时的动态状态数据使用 `Data` 容器进行管理。
- **性能红线 (Performance First)**:
  - 严禁在 `_Process` 或 `_PhysicsProcess` 等热路径中执行 `GetNode`、`new` 引用类型、字符串拼接或 LINQ 操作。
  - 对于高频生成的对象（如子弹、敌人、特效），必须使用 `ObjectPool`。

## 2. 代码规范与习惯

- **命名规范**:
  - **PascalCase**: 用于类名、方法名、公开属性以及 Godot 的输入动作名称（Input Actions）。
  - **\_camelCase**: 用于私有字段。
  - 脚本文件名必须与类名完全一致。
- **输入映射管理**:
  - 输入动作名称应直接在 `project.godot` 中定义为 PascalCase。
  - 使用 `InputDocGenerator` 工具自动从项目配置生成 `InputMap.md` 文档，严禁手动修改该文档。
- **日志记录**:
  - 每个类应声明一个静态的 `Log` 实例：`private static readonly Log _log = new Log("ClassName");`。
  - 使用分级日志，并利用 `Conditional("DEBUG")` 确保发布版本的零开销。

## 3. 组件系统特定规则

- **VelocityComponent (移动组件)**:
  - 应具备可选的“玩家输入控制”功能 (`EnablePlayerInput`)。
  - 当启用时，组件应能自动驱动父实体的移动，实现“添加即移动”的便捷体验。
- **解耦通信**:
  - 组件之间、组件与实体之间应通过 C# 事件 (`event Action`) 或 `Data` 容器的监听机制进行通信，减少直接耦合。

## 4. 自动化与文档

- **文档实时性**:
  - 关键系统（如 ECS 组件、输入映射）应配有配套的 `.md` 文档。
  - 尽可能利用工具（如 `InputDocGenerator`）保持文档与代码同步。
